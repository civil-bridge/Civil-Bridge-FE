# [BE 요청 사항] 제안서 투표 기능 오류 및 API 구조 개선 요청

현재 프론트엔드 연동 과정에서 제안서 투표 기능이 막혀서 투표가 전혀 되지 않는 가장 큰 문제가 있습니다 (400 Bad Request). 
이와 관련하여 단순히 비즈니스 로직(상태 값 검증 등) 버그일 수 있지만, 근본적으로 투표 가능 여부를 막고 있거나 실시간 현황 갱신을 어렵게 만드는 **API 설계 자체의 개선점**이 있어 다음과 같이 확인 및 개선을 요청드립니다. (투표 취소 기능은 설계에서 제외하는 현재 상태를 유지합니다.)

---

## 1. 투표 기능 불가(400 에러) 오류 해결 (최우선)
**현상:** 제안서를 제출한 본인뿐만 아니라 다른 사용자가(제 3자) 투표하기 버튼을 눌러도, 백엔드로부터 무조건 400 Bad Request가 돌아오면서 투표가 모두 거부되고 있습니다. 어제 점검 결과 프론트엔드 측의 API 호출(`POST /api/proposals/{proposalId}/consents`)은 명세에 맞게 정상적으로 이뤄지고 있습니다.

**요청 사항:**
현재 투표가 되어야 다음 프론트엔드의 화면 상태를 개발 및 테스트할 수 있습니다. 
과도한 방어 코드나 DB 상태 값(VOTING) 갱신 지연 등에 의한 이슈는 아닌지 백엔드의 `consentProposal()` 로직을 전반적으로 살펴봐 주세요.
지금의 화두는 **"왜 모든 투표가 튕기느냐"입니다. 작성자 본인 제약이나 룸 참여자 제한 등 백엔드의 유효성 검증에서 막히는 구간을 점검하고, 정상적으로 투표가 되도록 막힌 부분을 해결해 주시기 바랍니다.**

---

## 2. 실시간 투표수 현황 UI 반영을 위한 API 응답 구조 개선
**현상:** 투표 성공 시 프론트엔드에 투표된 "실시간 인원수"를 즉시 반영해야 합니다. 그런데 현재 명세서상의 `POST /consents` API 응답은 단순 성공 메시지뿐입니다.

```json
// 현재 응답 구조
{
  "code": "SUCCESS",
  "message": "제안서 동의 완료"
}
```
프론트엔드 입장에서는 투표 직후 동의 수가 몇 명으로 늘어났는지 알 수 없어, 이를 즉각 화면(현황)에 갱신하기 위해 투표자 목록이나 제안서 상태를 한 번 더 재조회(Refetch)해야 하는 비효율적인 상황이 발생합니다.

**요청 사항:**
실제로 쏟아지는 투표 행위에서, 사용자가 투표한 즉시 "실시간 인원 현황 수"가 부드럽게 화면에 렌더링될 수 있도록 **투표 성공 시 응답값(data)에 갱신된 `totalConsents(해당 제안서의 총 동의 수)` 를 함께 내려주는 쪽으로 API 데이터 설계 개선**을 강력히 요청합니다.

```json
// 개선 제안 응답
{
  "code": "SUCCESS",
  "message": "제안서 동의 완료",
  "data": {
    "totalConsents": 6  // 투표가 반영된 최신 동의 인원수
  }
}
```

이 부분의 API 설계가 개선되어야 프론트엔드에서 즉시 부드럽게 투표 인원수를 증가시켜 보여줄 수 있습니다. 이 2가지 위주로 점검 및 반영을 부탁드립니다.
