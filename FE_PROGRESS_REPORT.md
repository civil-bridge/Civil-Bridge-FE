# Civil Bridge 프론트엔드 진행 상황 보고서

현재(2026-03-04 16:39:22 KST 기준) Civil Bridge 프론트엔드 애플리케이션의 구현 상태, API 연동 현황 및 정상 동작 플로우, 그리고 아직 미완성인 부분들을 정리한 문서입니다.

## 1. 구현 완료 메인 플로우 및 API 연동 현황

현재 프론트엔드는 크게 인증, 메인 페이지(방 목록), 방 내부(채팅, 제안서) 플로우로 구현되어 있으며, 성공적으로 연동된 API들은 다음과 같습니다.

### 1-1. 회원가입 및 로그인 플로우
- **API 연동**: `auth.ts` (로그인, 로그아웃, 토큰 갱신), `user.ts` (회원가입, 이메일 인증 발송, 이메일 인증 확인)
- **정상 동작 플로우**:
  1. `/signup` 페이지에서 이메일 인증 번호 발송 및 확인 완료 후 회원가입 진행 (`user.ts` API 모두 정상 연동)
  2. `/login` 페이지에서 이메일/비밀번호 입력 시 로그인 성공 (`auth.ts`의 `login` API 연동 완료)
  3. 로그인 성공 시 리프레시 토큰(쿠키) 및 액세스 토큰(상태)를 기반으로 사용자 세션 유지
  4. 로그아웃 수행 가능 (`auth.ts`의 `logout` API 연동 완료)
  5. **액세스 토큰 만료 시**: `axios.ts`의 인터셉터를 통해 리프레시 토큰으로 자동 토큰 재발급 로직 구현 완료

### 1-2. 방(Room) 생성 및 목록 조회 플로우
- **API 연동**: `room.ts` (방 생성, 전체 방 목록 조회, 내 방 목록 조회)
- **정상 동작 플로우**:
  1. 메인 페이지(`/`) 접속 시 전체 방 목록 페이징 조회 완료 (`getTotalRooms` API 연동 완료) 
  2. 필터(Access Level) 등에 따른 방 목록 렌더링 정상 동작
  3. 새로운 방 생성 모달에서 지역, 주제, 제목 등을 입력하여 방 생성 정상 동작 (`createRoom` API 연동 완료)
  4. '내 참여 방' 탭이나 마이룸 페이지(`/my-rooms`) 진입 시 내가 참여중인 방 목록 조회 정상 동작 (`getMyJoinedRooms` API 연동 완료)

### 1-3. 방 진입 및 상호작용 (채팅 및 참가자)
- **API 연동**: `room.ts` (방 입장, 방 정보 조회, 방 퇴장, 방 삭제), `message.ts` (이전 메시지 조회)
- **정상 동작 플로우**:
  1. 방 진입 시 `joinRoom` API를 호출하여 방 참가 및 상세 정보(참가자 목록 등) 획득
  2. 방 내부 진입 시 `getMessagesByRoom` API를 통해 이전 채팅 기록 페이징(커서 기반) 조회 완료
  3. WebSocket(STOMP)을 통한 실시간 채팅 전송 및 수신 정상 동작
  4. 우측 상단의 참가자 목록 렌더링 정상 동작 (기본 10명 표시 후 '전체보기' 클릭 시 전체 모달 렌더링)
  5. 방장인 경우 '방 삭제', 일반 참가자인 경우 '방 나가기' 버튼 동작 연동 완료 (`deleteRoom`, `leaveRoom` API)

### 1-4. 제안서(Proposal) 플로우
- **API 연동**: `proposal.ts` (제안서 생성, 방별 제안서 목록 조회, 특정 제안서 조회, 제안서 수정, 락 상태 조회, 편집 시작/종료, 투표 시작, 투표 결과 확정, 투표 동의, 동의자 목록 조회)
- **정상 동작 플로우**:
  1. 방 우측 슬라이드 아웃 메뉴에서 제안서 목록 조회 및 상태(작성중, 투표 중, 가결됨 등) 뱃지 렌더링 완료 (`getProposalsByRoom`)
  2. **동시성 제어(Locking)**: 제안서 수정 시 `startEditing`을 호출하여 락을 획득하고, 다른 사용자는 수정 불가능하도록 UI 블락 처리 및 락 소유자 표시 완료 (`getLockStatus`)
  3. 락 소유자가 작성을 마치면 `finishEditing` 호출 및 락 해제
  4. 제안서 신규 생성 가능 (`createProposal`), 기존 제안서 내용 업데이트 가능 (`updateProposal`)
  5. 제안서 제출 조건 충족(UNSUBMITTABLE → SUBMITTABLE 논리) 시 방장이 **투표 시작(startVoting)** 가능 (최소 동의자 수 설정 등 포함)
  6. **투표 진행(VOTING)**: 참가자들이 '동의하기' 버튼을 눌러 동의 진행 (`consentProposal`). 정족수 대비 현재 동의 인원수를 시각적으로 보여주는 **게이지 바(Progress Bar)** 및 **마감 기한/남은 시간 표시 로직** 렌더링 완료.
  7. **가결/확정(COMPLETED/REJECTED)**: 기한이 만료되거나 정족수를 채웠을 때 방장이 '투표 결과 확정하기' 버튼을 눌러 상태 확정 로직 완료 (`endVoting`)
  8. 가결된 제안서 클릭 시 해당 지역 민원 사이트(국민신문고 등)로 이동하는 URL 아웃링크 버튼 렌더링 완료


## 2. 미완성 기능 및 향후 개발 필요 사항 (To-Do)

현재 프론트엔드 단에서 추가 구현이나 백엔드와 협의가 필요한 사항들입니다.

### 2-1. UI/UX 및 파일/에셋 관련
- [ ] **이미지/파일 업로드**: 제안서 작성이나 채팅 시스템 내 파일 혹은 이미지 첨부 기능 미구현 (API 부재)
- [ ] **에러바운더리 및 글로벌 로딩 상태 핸들링**: Axios 에러 처리 일부(인터셉터)는 되었으나, 페이지 레벨의 우아한 에러 핸들링 및 스켈레톤 UI(로딩스피너) 적용 고도화 필요
- [ ] **반응형 웹 최적화**: 모바일 사이즈에서의 사이드바 및 레이아웃 디테일 조율 필요 (일부 모달 겹침 현상 가능성 체크)

### 2-2. 기능 및 API 추가 필요 사항
- [ ] **사용자 프로필 관리**: 닉네임 변경, 내 정보 조회(마이페이지 상세 내역) 등에 대한 API 연동 미구현
- [ ] **알림 시스템(Notification)**: 내 방에 새로운 제안서가 등록되거나 가결되었을 때 등에 대한 실시간 푸시 알림 로직 부재
- [ ] **실제 민원 사이트 지역별 URL 동적 바인딩**: 현재 가결 상태일 때 이동하는 URL이 국민신문고 하나로 고정되어 있음. DB 혹은 서버에서 각 지역에 맞는 정확한 url을 내려줄 예정이라면 DTO에 필드 추가 및 통신 방식 업데이트 필요
- [ ] **웹소켓(STOMP) 연결 시 토큰 인증 상세 처리**: 현재 프론트에서 Stompjs 헤더에 토큰은 넣고 있으나, 토큰 만료 시 소켓 재연결(Refresh Token 핸들링) 흐름에 대한 안정화 테스트 필요
